# RFC-0001: Origin Protocol v1 Specification

**Status:** Draft  
**Category:** Standards Track  
**Author:** Origin Protocol Team  
**Created:** 2026-06-08  
**Updated:** 2026-06-08 (v1.1 — type field, revocation)

## Abstract

Origin is a cryptographic provenance protocol that binds a public key, a timestamp, and a digital artifact hash in a self-contained signed statement. The protocol defines a canonical text format for provenance statements, a deterministic signing procedure, a verification algorithm requiring no network access or external infrastructure, and a revocation mechanism for compromised keys.

## 1. Notation

- `||` denotes byte concatenation.
- `0x` prefix denotes hexadecimal encoding.
- `b64(·)` denotes base64url encoding per RFC 4648 §5 WITH padding.
- `sha256(·)` denotes the SHA-256 hash function (FIPS 180-4).
- Ed25519 refers to the Edwards-curve Digital Signature Algorithm per RFC 8032.

## 2. Primitive Types

### 2.1 Artifact Hash

An artifact hash is a hash digest of an artifact, encoded as a lowercase hexadecimal string prefixed with an algorithm identifier.

**Format:** `<algorithm>:<lowercase hex chars>`  
**Supported algorithms:**

| Algorithm | Prefix | Hex length | Binary size |
|---|---|---|---|
| SHA-256 | `sha256` | 64 | 32 bytes |
| SHA-384 | `sha384` | 96 | 48 bytes |
| SHA-512 | `sha512` | 128 | 64 bytes |

**Example:** `sha256:ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`

### 2.2 Public Key

An Ed25519 public key is a 32-byte compressed point encoding, encoded as 44 base64url characters (with padding).

**Format:** 44 base64url characters (RFC 4648 §5, with `=` padding)  
**Binary size:** 32 bytes  
**Encoded size:** 44 ASCII characters

### 2.3 Signature

An Ed25519 signature is a 64-byte (R ‖ S) encoding, encoded as 88 base64url characters (with padding).

**Format:** 88 base64url characters (RFC 4648 §5, with `=` padding)  
**Binary size:** 64 bytes  
**Encoded size:** 88 ASCII characters

### 2.4 Timestamp

A timestamp is a non-negative decimal integer representing seconds since the UNIX epoch (1970-01-01T00:00:00Z).

**Constraint:** `0 ≤ timestamp ≤ 253402300799` (year 9999)  
**Format:** Decimal ASCII digits, no leading zeros (except the value `0` itself)

Timestamps are self-asserted by the signer. They are format-validated but NOT covered by the signature. See §9.3.

### 2.5 Statement Type

Every statement carries a `type` field that identifies its purpose:

| Type | Purpose |
|---|---|
| `provenance` | Binds an artifact hash to a public key with a timestamp |
| `revocation` | Declares that a compromised key's statements should no longer be trusted |

### 2.6 Protocol Version

The protocol version is always `v1`.

## 3. Statement Format

All statements are UTF-8 encoded text files with LF (`\n`, `0x0a`) line endings, containing colon-separated key-value pairs, one per line.

### 3.1 Provenance Statement (no parent)

```
origin: v1
type: provenance
hash: <algorithm>:<hex>
time: <decimal timestamp>
key: <44 base64url chars>
sig: <88 base64url chars>
```

### 3.2 Provenance Statement (with parent)

```
origin: v1
type: provenance
parent: <algorithm>:<hex>
hash: <algorithm>:<hex>
time: <decimal timestamp>
key: <44 base64url chars>
sig: <88 base64url chars>
```

### 3.3 Revocation Statement

```
origin: v1
type: revocation
revoked: <44 base64url chars>
since: <decimal timestamp>
key: <44 base64url chars>
sig: <88 base64url chars>
```

| Key | Value Constraint |
|---|---|
| `origin` | Exactly `v1` |
| `type` | Exactly `provenance` or `revocation` |
| `parent` | Same format as `hash` (provenance only) |
| `hash` | `<algorithm>:<hex>` matching §2.1 (provenance only) |
| `revoked` | Exactly 44 base64url chars — the key being revoked (revocation only) |
| `time` | ASCII decimal integer, no leading zeros (except `0`), range [0, 253402300799] (provenance only) |
| `since` | Same format as `time` — statements by the revoked key with `time >= since` are not trusted (revocation only) |
| `key` | Exactly 44 base64url characters (RFC 4648 §5 with padding) |
| `sig` | Exactly 88 base64url characters (RFC 4648 §5 with padding) |

### 3.4 Line Format Rules

1. Each line matches the pattern `<key>: <value>`, where `<key>` is an ASCII lowercase alphabetic string and `<value>` is a UTF-8 string with no leading or trailing whitespace.
2. The separator is exactly `: ` (colon followed by one ASCII space, `0x3a 0x20`).
3. Each line is terminated by `\n` (LF, `0x0a`).
4. The file ends with `\n` (the last line is terminated, making the file end with LF).

### 3.5 Structural Constraints

1. **Line count:** 6 lines for provenance without parent, 7 lines for provenance with parent, 6 lines for revocation.
2. **Line order:** Strict ordering per the formats above. No reordering permitted.
3. **No duplicate keys.** Each key appears exactly once.
4. **No unknown keys.** Only the defined keys are permitted.
5. **No BOM** (Byte Order Mark, `0xef 0xbb 0xbf`).
6. **No CR** (`\r`, `0x0d`) anywhere in the file.
7. **No null bytes** (`0x00`) anywhere in the file.
8. **No control characters** (code points U+0000–U+001F and U+007F) in values, except the LF line terminators.

## 4. Canonical Body

The canonical body is the input to the Ed25519 signing and verification operations.

### 4.1 Provenance (no parent)

```
origin: v1
type: provenance
hash: <hex>
key: <b64>
```

Lines: origin, type, hash, key. No time, no sig, no trailing newline.

### 4.2 Provenance (with parent)

```
origin: v1
type: provenance
parent: <hex>
hash: <hex>
key: <b64>
```

Lines: origin, type, parent, hash, key. No time, no sig, no trailing newline.

### 4.3 Revocation

```
origin: v1
type: revocation
revoked: <b64>
since: <ts>
key: <b64>
```

Lines: origin, type, revoked, since, key. No sig, no trailing newline.

## 5. Signing Procedure

### 5.1 Provenance Signing

**Inputs:**
- Ed25519 secret key `sk` (32 bytes)
- Artifact bytes `data` (arbitrary length)
- Timestamp `ts` (non-negative integer)
- Optional parent hash `parent` (if chaining)

**Algorithm:**
```
1. hash = sha256(data)
2. hex_hash = lowercase_hex(hash)           // 64 hex chars
3. Encoded fields:
   - origin_line = "origin: v1"
   - type_line = "type: provenance"
   - parent_line = "parent: <parent>"       // if parent present
   - hash_line = "hash: sha256:" || hex_hash
   - key_line = "key: " || b64(pubkey(sk))
4. canonical = join(lines, "\n")            // no trailing newline
5. sig = ed25519_sign(sk, canonical)
6. Append "sig: " || b64(sig) || "\n"
7. return full statement
```

### 5.2 Revocation Signing

**Inputs:**
- Ed25519 secret key `sk` (32 bytes)
- Base64-encoded revoked public key `revoked_b64`
- Timestamp `since`
- Base64-encoded signer public key `signer_b64`

**Algorithm:**
```
1. Encoded fields:
   - origin_line = "origin: v1"
   - type_line = "type: revocation"
   - revoked_line = "revoked: " || revoked_b64
   - since_line = "since: " || decimal(since)
   - key_line = "key: " || signer_b64
2. canonical = join(lines, "\n")            // no trailing newline
3. sig = ed25519_sign(sk, canonical)
4. Append "sig: " || b64(sig) || "\n"
5. return full revocation statement
```

## 6. Verification Procedure

### 6.1 Provenance Verification

**Inputs:**
- Statement bytes
- Artifact bytes `data`

**Algorithm:**
```
1. Parse statement → 6 or 7 fields
2. Validate all field constraints (§3.5)
3. Verify type == "provenance"
4. Reconstruct canonical body (§4.1 or §4.2)
5. Decode sig (base64url → 64 bytes)
6. Decode key (base64url → 32 bytes)
7. actual_hash = <algorithm>(data) per hash prefix
8. If actual_hash ≠ parsed hash: FAILED (hash mismatch)
9. If ed25519_verify(key, canonical, sig) fails: FAILED (signature mismatch)
10. Return VERIFIED
```

### 6.2 Revocation Verification

**Inputs:**
- Revocation statement bytes

**Algorithm:**
```
1. Parse statement → 6 fields
2. Validate all field constraints (§3.5)
3. Verify type == "revocation"
4. Reconstruct canonical body (§4.3)
5. Decode sig
6. Decode key
7. If ed25519_verify(key, canonical, sig) fails: FAILED
8. Return VERIFIED
```

### 6.3 Verification Results

| Result | Meaning |
|---|---|
| `VERIFIED` | The statement is cryptographically valid |
| `FAILED` | The statement is invalid — see error message |

## 7. Key Generation

Ed25519 key pairs are generated per RFC 8032 §5.1.5:

1. Generate 32 cryptographically secure random bytes as the seed.
2. Hash the seed with SHA-512.
3. Prune the first 32 bytes: clear bits 0–3, clear bit 254, set bit 255.
4. The secret key is the original 32-byte seed.
5. The public key is the encoded scalar multiplication of the pruned hash with the basepoint.

## 8. Audit Output

The audit function parses a statement and displays its fields in human-readable form.

**Provenance output example:**
```
Statement Audit
├─ Type:    provenance
├─ Origin:  v1
├─ Parent:  sha256:<hex>               (if present)
├─ Hash:    sha256:<hex> (SHA-256)
├─ Time:    2024-06-07T16:00:00Z (1717776000) — advisory
├─ Key:     <base64url>

```

**Revocation output example:**
```

├─ Type:    revocation
├─ Origin:  v1
├─ Revoked: <base64url> (key)
├─ Since:   2024-06-07T16:00:00Z (1717776000)

```

## 9. Security Considerations

### 9.1 Cryptographic Assumptions

- Ed25519 provides existential unforgeability under chosen-message attack (EUF-CMA).
- SHA-256 provides collision resistance (≤ 2^128 operations).
- The 256-bit Ed25519 scalar field provides 128-bit security against discrete log attacks.

### 9.2 Hash Agility

The protocol supports SHA-256, SHA-384, and SHA-512. This allows users to choose their security margin. SHA-256 is the default. All supported algorithms are FIPS 180-4 compliant.

### 9.3 Timestamp Advisory

Timestamps are self-asserted by the signer. The verifier only checks that the timestamp is a well-formed non-negative integer within an acceptable range. There is no mechanism to verify that the timestamp reflects "real" time. The timestamp is NOT included in the canonical body and changing it does NOT invalidate the signature.

### 9.4 Revocation

Revocation statements are self-issued statements signed by the key being revoked (or a designated recovery key). They declare that all provenance statements signed by the revoked key with `time >= since` should no longer be trusted. Revocation does not require network access or external infrastructure.

### 9.5 Key Compromise

A compromised signing key cannot be revoked retroactively at the protocol level. Revocation statements can only signal that future statements should not be trusted. Statements signed before the compromise but after the `since` timestamp remain suspect.

### 9.6 Side-Channel Resistance

The reference implementation uses `ed25519-dalek`, which implements constant-time point operations. However, the environment (operating system, hardware) may introduce timing variability outside the library's control.

## 10. CLI Reference

### `origin hash <path>`
Compute the hash of a file.

### `origin keygen [--output <dir>]`
Generate a new Ed25519 key pair. Secret key permissions set to 0o600 on Unix.

### `origin sign <path> [--key <file>] [--time <ts>] [--parent <path>]`
Sign an artifact, producing a `.origin` statement file. Uses current time by default.
Secret key from `--key <file>`, `ORIGIN_KEY` env var, or `--key -` (stdin).

### `origin verify <statement> <artifact>`
Verify a provenance statement against an artifact. Returns VERIFIED or FAILED.

### `origin audit <statement>`
Display a human-readable audit of any statement (provenance or revocation).

### `origin revoke --key <file> --revoked <b64> [--since <ts>]`
Create a revocation statement for a compromised key. Output written to `revocation-<prefix>.origin`.

### `origin check <statement> --revocation <file>`
Check whether a provenance statement is revoked by a given revocation statement.
