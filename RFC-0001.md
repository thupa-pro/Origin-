# RFC-0001: Origin Protocol v1 Specification

**Status:** Draft  
**Category:** Standards Track  
**Author:** Origin Protocol Team  
**Created:** 2026-06-08

## Abstract

Origin is a cryptographic provenance protocol that binds a public key, a timestamp, and a digital artifact hash in a self-contained signed statement. The protocol defines a canonical text format for provenance statements, a deterministic signing procedure, and a verification algorithm requiring no network access or external infrastructure.

## 1. Notation

- `||` denotes byte concatenation.
- `0x` prefix denotes hexadecimal encoding.
- `b64(·)` denotes base64url encoding per RFC 4648 §5 WITH padding.
- `sha256(·)` denotes the SHA-256 hash function (FIPS 180-4).
- Ed25519 refers to the Edwards-curve Digital Signature Algorithm per RFC 8032.

## 2. Primitive Types

### 2.1 Artifact Hash

An artifact hash is a 32-byte SHA-256 digest, encoded as 64 lowercase hexadecimal characters prefixed with `sha256:`.

**Format:** `sha256:<64 lowercase hex chars>`  
**Example:** `sha256:ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`

### 2.2 Public Key

An Ed25519 public key is a 32-byte compressed point encoding, encoded as 44 base64url characters (with padding).

**Format:** 44 base64url characters (RFC 4648 §5, with `=` padding)  
**Binary size:** 32 bytes  
**Encoded size:** 44 ASCII characters

### 2.3 Signature

An Ed25519 signature is a 64-byte (R ‖ S) encoding, encoded as 88 base64url characters (with padding).

**Format:** 88 base64url characters (RFC 4648 §5, with `=` padding)  
**Binary size:** 64 bytes  
**Encoded size:** 88 ASCII characters

### 2.4 Timestamp

A timestamp is a non-negative decimal integer representing seconds since the UNIX epoch (1970-01-01T00:00:00Z).

**Constraint:** `0 ≤ timestamp ≤ 253402300799` (year 9999)  
**Format:** Decimal ASCII digits, no leading zeros (except the value `0` itself)

### 2.5 Protocol Version

The protocol version is always `v1`.

## 3. Statement Format

A statement is a UTF-8 encoded text file with LF (`\n`, `0x0a`) line endings, containing exactly five colon-separated key-value pairs, one per line.

```
origin: v1
hash: sha256:<64 hex chars>
time: <decimal timestamp>
key: <44 base64url chars>
sig: <88 base64url chars>
```

### 3.1 Line Format Rules

1. Each line matches the pattern `<key>: <value>`, where `<key>` is an ASCII lowercase alphabetic string and `<value>` is a UTF-8 string with no leading or trailing whitespace.
2. The separator is exactly `: ` (colon followed by one ASCII space, `0x3a 0x20`).
3. Each line is terminated by `\n` (LF, `0x0a`).
4. The file ends with `\n` (the 5th line is terminated, making the file end with LF).

### 3.2 Structural Constraints

1. **Exactly 5 lines** must be present. Empty lines are rejected.
2. **Line order** must be: `origin`, `hash`, `time`, `key`, `sig`.
3. **No duplicate keys**. Each key appears exactly once.
4. **No unknown keys**. Only the 5 defined keys are permitted.
5. **No BOM** (Byte Order Mark, `0xef 0xbb 0xbf`).
6. **No CR** (`\r`, `0x0d`) anywhere in the file.
7. **No null bytes** (`0x00`) anywhere in the file.
8. **No control characters** (code points U+0000–U+001F and U+007F) in values, except the LF line terminators.

### 3.3 Field Constraints

| Key | Value Constraint |
|---|---|
| `origin` | Exactly `v1` |
| `hash` | Exactly `sha256:` followed by 64 lowercase hex chars (`[0-9a-f]{64}`) |
| `time` | ASCII decimal integer, no leading zeros (except `0`), range [0, 253402300799] |
| `key` | Exactly 44 base64url characters (RFC 4648 §5 with padding) |
| `sig` | Exactly 88 base64url characters (RFC 4648 §5 with padding) |

## 4. Canonical Body

The canonical body is the input to the Ed25519 signing and verification operations.

### 4.1 Definition

The canonical body is constructed from the first four fields of the statement (excluding `sig`) in their exact line format, joined by `\n`, with NO trailing newline:

```
origin: v1
hash: sha256:<64 hex chars>
time: <decimal timestamp>
key: <44 base64url chars>
```

This is byte-for-byte identical to the content of lines 1–4 of the statement file with the trailing `\n` of line 4 removed.

### 4.2 Construction Algorithm

```
let lines = [
    "origin: v1",
    "hash: sha256:<hex>",
    "time: <ts>",
    "key: <b64>",
]
return join(lines, "\n")  // no trailing newline
```

## 5. Signing Procedure

### 5.1 Inputs

- Ed25519 secret key `sk` (32 bytes)
- Artifact bytes `data` (arbitrary length)
- Timestamp `ts` (non-negative integer)
- Protocol version `"v1"` (constant)

### 5.2 Algorithm

```
1. hash = sha256(data)
2. hex_hash = lowercase_hex(hash)           // 64 hex chars
3. Encoded fields:
   - origin_line = "origin: v1"
   - hash_line = "hash: sha256:" || hex_hash
   - time_line = "time: " || decimal(ts)
   - key_line = "key: " || b64(pubkey(sk))
4. canonical = join([origin_line, hash_line, time_line, key_line], "\n")
5. sig = ed25519_sign(sk, canonical)
6. statement = canonical || "\n" || "sig: " || b64(sig) || "\n"
7. return statement
```

### 5.3 Output

The output is a UTF-8 encoded byte sequence (the statement file content).

## 6. Verification Procedure

### 6.1 Inputs

- Statement bytes (the complete statement file content)
- Artifact bytes `data`

### 6.2 Algorithm

```
1. Parse `statement` into 5 fields (see §3).
2. Validate all field constraints (see §3.3).
3. Extract `sig` from parsed fields.
4. Extract `key` from parsed fields.
5. Extract `time` from parsed fields (convert to integer).
6. Extract `hash` from parsed fields (strip "sha256:" prefix).
7. Reconstruct canonical body from first 4 fields (§4.2).
8. Decode `sig` from base64url to 64 bytes.
9. Decode `key` from base64url to 32 bytes.
10. actual_hash = sha256(data)
11. If actual_hash ≠ parsed hash: return FAILED (hash mismatch)
12. If ed25519_verify(key, canonical, sig) fails: return FAILED (signature mismatch)
13. Return VERIFIED
```

### 6.3 Verification Results

| Result | Meaning |
|---|---|
| `VERIFIED` | The statement is cryptographically valid for the given artifact |
| `FAILED` | The statement is invalid — see error message |

The verification procedure MUST NOT return any third state (e.g., "unknown", "untrusted", "expired").

## 7. Key Generation

Ed25519 key pairs are generated per RFC 8032 §5.1.5:

1. Generate 32 cryptographically secure random bytes as the seed.
2. Hash the seed with SHA-512.
3. Prune the first 32 bytes: clear bits 0–3, clear bit 254, set bit 255.
4. The secret key is the original 32-byte seed.
5. The public key is the encoded scalar multiplication of the pruned hash with the basepoint.

## 8. Audit Output

The audit function parses a statement and displays its fields in human-readable form. It does NOT perform cryptographic verification. It is a diagnostic tool.

**Output format:**

```
Statement: <path>
  Origin:  v1
  Hash:    sha256:<hex>
  Time:    <ISO 8601> (<decimal>)
  Key:     <base64url>
  Sig:     <base64url>
  Verdict: VERIFIED | FAILED | NOT CHECKED
```

## 9. Security Considerations

### 9.1 Cryptographic Assumptions

The protocol is secure under the following standard cryptographic assumptions:

- Ed25519 provides existential unforgeability under chosen-message attack (EUF-CMA).
- SHA-256 provides collision resistance (≤ 2^128 operations).
- The 256-bit Ed25519 scalar field provides 128-bit security against discrete log attacks.

### 9.2 Side-Channel Resistance

The reference implementation uses `ed25519-dalek`, which implements constant-time point operations. However, the environment (operating system, hardware) may introduce timing variability outside the library's control.

### 9.3 TOCTOU

See threat model (THREAT_MODEL.md). Verification is a snapshot operation of caller-provided byte buffers. File-system-level verification is a convenience wrapper with documented race conditions.
